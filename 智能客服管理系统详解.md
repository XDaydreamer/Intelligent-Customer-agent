## 1. 项目概述

### 1.1 项目定位

这是一个基于 **LangGraph** 框架的**电商智能客服系统**，支持以下核心功能：

| 模块 | 功能简述 |
|------|---------|
| 智能对话 | 多轮对话客服，带知识库检索增强生成（RAG）和会话记忆 |
| 知识库管理 | 创建/管理知识库，上传文档（txt/json/docx/csv/xlsx），自动分块向量化 |
| 文档知识 | 根据产品信息自动生成营销文案（AI Copywriting） |
| 知能对话 | 预设问答对管理，支持树形分组，用于快速精确回复 |
| 聊天记录 | 历史会话查看、AI 回复编辑、选择性保存为预设对话 |
| 客服模版 | 客服回复模板管理，注入到对话 Agent 的系统提示中 |
| 转人工设置 | 关键词触发规则，匹配后自动回复并支持人工转接 |
| 商店配置 | 预留模块（开发中） |

## 4. 后端详解

### 4.1 FastAPI 应用入口 (`main.py`)

应用使用 **lifespan 上下文管理器** 管理生命周期：

- **启动时**：调用 `init_db()` 自动建表，启动 `periodic_cleanup()` 后台任务
- **运行时**：注册 6 个路由模块，CORS 中间件允许 localhost:3000
- **关闭时**：取消清理任务

```python
@asynccontextmanager
async def lifespan(app: FastAPI):
    await init_db()
    cleanup_task = asyncio.create_task(periodic_cleanup())  # 每10分钟清理过期会话
    yield
    cleanup_task.cancel()
```

### 4.2 配置系统 (`src/config.py`)

使用 `pydantic-settings` 的 `BaseSettings`，自动从 `.env` 文件读取环境变量。

关键配置项：

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `llm_model` | `qwen-plus` | 千问模型名 |
| `embedding_model` | `text-embedding-v2` | 嵌入模型名 |
| `dashscope_base_url` | `https://dashscope.aliyuncs.com/compatible-mode/v1` | 阿里云百炼端点 |
| `max_messages_per_session` | 40 | 触发压缩的消息阈值 |
| `recent_messages_keep` | 20 | 压缩后保留的最近消息数 |
| `session_ttl_days` | 10 | 会话数据保留天数 |
| `chroma_persist_dir` | `./chroma_data` | ChromaDB 本地存储路径 |

### 4.3 服务层 (`src/services/`)

#### KnowledgeService — 知识库管理 + RAG 摄入

| 方法 | 说明 |
|------|------|
| `create(data)` | 创建知识库 + 对应的 ChromaDB 集合 |
| `upload_file(kb_id, filename, content)` | 文件解析 → 分块 → 向量化 → 入库 |
| `_chunk_text(text, 500, 50)` | 500 字符块 + 50 字符重叠的滑动窗口 |
| `add_text_to_kb(kb_id, text, filename)` | 纯文本直接向量化入库 |
| `delete(kb_id)` | 同时删除 PostgreSQL 记录和 ChromaDB 集合 |

#### MemoryService — 会话记忆管理（核心）

| 方法 | 说明 |
|------|------|
| `create_conversation()` | 创建新会话，自动分配"客户N"编号 |
| `add_message(conv_id, role, content)` | 添加消息，更新计数和标题 |
| `load_history(conv_id)` | 返回 (摘要, 未压缩消息列表) 供 Agent 使用 |
| `maybe_compress(conv_id)` | 超过 40 条消息时触发 LLM 压缩 |
| `_summarize_messages()` | 调用 LLM 将旧消息压缩为 ≤200 字摘要 |
| `cleanup_expired()` | 删除 10 天前的过期会话 |
| `update_message(msg_id, content)` | 编辑消息内容 |
| `create_preset_dialog(q, a, parent_id)` | 保存问答对为预设对话（带去重） |

#### DialogService — 预设对话匹配

- `search(query)` — 在 `PresetDialog.question` 上执行 `ILIKE '%query%'` 子串匹配
- 完全匹配时跳过 LLM 直接返回预设回复（性能优化）

### 4.4 LLM 工厂 (`src/agents/llm.py`)

```python
def get_llm(temperature=0.3) -> ChatOpenAI:
    return ChatOpenAI(
        model=settings.llm_model,          # qwen-plus
        api_key=settings.dashscope_api_key,
        base_url=settings.dashscope_base_url,  # DashScope 兼容端点
        temperature=temperature,
    )

def get_embeddings() -> OpenAIEmbeddings:
    return OpenAIEmbeddings(
        model=settings.embedding_model,    # text-embedding-v2
        api_key=settings.dashscope_api_key,
        base_url=settings.dashscope_base_url,
    )
```

关键：使用 `langchain-openai` 的 `ChatOpenAI` 类，通过 `base_url` 指向 DashScope 的 OpenAI 兼容端点，实现对千问模型的调用。无需额外的 DashScope SDK。

### 5.3 状态管理

项目使用**纯 React 本地状态**，无 Redux/MobX/Zustand：

- **跨组件共享**：`selectedKbId` 通过 props drilling 从 App → KnowledgeNewPage → KnowledgeUploadPage
- **跨会话持久化**：`conversation_id` 通过 `localStorage` 存储
- **服务端状态**：各页面组件通过 `useEffect` + `fetch` 加载数据，无缓存层

### 5.4 API 层 (`services/api.ts`)

统一的 `request<T>()` 函数处理所有 HTTP 请求：

```typescript
async function request<T>(url: string, options?: RequestInit): Promise<T> {
  const res = await fetch(`/api${url}`, {
    headers: { 'Content-Type': 'application/json' },
    ...options,
  });
  if (!res.ok) throw new Error(await parseError(res));
  if (res.status === 204) return undefined;
  return res.json();
}
```

共封装了 **28 个 API 调用**，分为 6 组：`knowledgeApi`、`templateApi`、`dialogApi`、`transferRuleApi`、`chatApi`、`conversationApi`、`copywritingApi`。

|----|------|------|
| `id` | VARCHAR(36) PK | UUID |
| `conversation_id` | VARCHAR(36) FK | 所属会话 |
| `role` | VARCHAR(20) | 'user' / 'assistant' |
| `content` | TEXT | 消息内容 |
| `is_summarized` | BOOLEAN | 是否已被压缩 |
| `token_count` | INTEGER | 估算 Token 数 |
| `created_at` | TIMESTAMPTZ | 创建时间 |

### 6.2 ChromaDB 向量存储

- **客户端类型**：`PersistentClient`（本地文件持久化，目录 `./chroma_data`）
- **集合命名**：每个知识库对应一个集合，命名为 `kb_{knowledge_base_id}`
- **文档块**：以 `{file_id}_chunk_{i}` 为 ID 存储，元数据包含 `source` 和 `kb_id`

## 7. 记忆系统设计

### 7.1 设计思路

记忆系统围绕**会话（Conversation）**概念设计。每个会话代表一个独立客户的完整咨询过程。

### 7.2 会话生命周期

```
客户首次发消息
  │
  ▼
MemoryService.create_conversation()
  │  自动分配 customer_label: "客户N"
  │  title = None（将在第一条消息时设置）
  │
  ▼
每次对话轮次：
  MemoryService.add_message(user_msg)
  MemoryService.add_message(assistant_msg)
  │  message_count += 1
  │  title ← 首条用户消息前 80 字符
  │  updated_at ← 当前时间
  │
  ▼
每次对话后检查压缩：
  MemoryService.maybe_compress()
  │  未压缩消息 > 40?
  │  ├─ YES → 取最旧的 N-20 条消息 → LLM 生成摘要
  │  │        标记 is_summarized=True
  │  │        更新 conv.summary
  │  └─ NO  → 跳过
  │
  ▼
后台定时任务（每10分钟）：
  MemoryService.cleanup_expired()
  │  删除 updated_at < NOW() - 10天 的会话
```

### 7.3 压缩算法

```
触发条件：未压缩消息数 > 40

步骤：
1. 计算需要压缩的消息数：excess = 未压缩数 - 20
2. 取出最旧的 excess 条消息
3. 调用 LLM (temperature=0.1) 生成摘要：
   - 提示词要求保留客户身份、需求、偏好、问题等关键信息
   - 摘要控制在 200 字以内
   - 若已有旧摘要，则与新内容合并
4. 将旧消息标记为 is_summarized=True
5. 更新 conv.summary = 新摘要
```

### 7.4 上下文注入

当 Agent 生成回复时，上下文按以下顺序组装：

```
[System Prompt]      ← 角色设定 + 模版内容 + 知识库资料
[对话摘要]           ← conv.summary（若存在），以 system 角色注入
[最近未压缩消息]     ← 按时间升序，最多 40 条
[当前用户消息]       ← 最新 HumanMessage
```

### 7.5 过期清理

- **触发方式**：uvicorn 后台任务，每 10 分钟自动执行
- **清理逻辑**：`DELETE FROM conversations WHERE updated_at < NOW() - INTERVAL '10 days'`
- **级联删除**：消息表设置了 `ON DELETE CASCADE`，会话删除时消息自动清除

## 8. RAG 工程

RAG（Retrieval-Augmented Generation）是本系统的核心能力，让 Agent 能够基于上传的知识库文档回答客户问题。

### 8.1 文档摄入管道

```
用户上传文件
  │
  ▼
1. 保存原始文件到 ./uploads/{uuid}.{ext}
  │
  ▼
2. 解析文件为纯文本
   ├─ .txt/.md  → UTF-8 解码
   ├─ .json     → 解析并美化打印
   ├─ .docx     → python-docx 段落提取
   ├─ .csv      → pandas.read_csv → to_string
   └─ .xlsx     → pandas.read_excel → to_string
  │
  ▼
3. 文本分块（_chunk_text）
   块大小：500 字符
   重叠：50 字符
   方式：简单滑动窗口
  │
  ▼
4. 向量化并存储到 ChromaDB
   集合名：kb_{knowledge_base_id}
   块 ID：{file_id}_chunk_0, _chunk_1, ...
   元数据：{source: filename, kb_id: kb_id}
  │
  ▼
5. 在 PostgreSQL 记录 KnowledgeDocument
   (id, kb_id, filename, file_type, chunk_count)
```

### 8.2 检索流程（对话时）

```
用户消息："这件衣服掉色怎么办"
  │
  ▼
1. ChromaDB 查询
   collection.query(
     query_texts=["这件衣服掉色怎么办"],
     n_results=5
   )
  │
  ▼
2. 返回相关文档块：
   [
     "如果衣物出现染色/掉色问题，建议首次洗涤时...",
     "所有面料均通过色牢度检测...",
     ...
   ]
  │
  ▼
3. 注入 LLM 上下文：
   System Prompt += "## 知识库参考资料:\n" + docs_text
  │
  ▼
4. LLM 基于参考文档生成回复
```

### 8.3 当前局限

- **嵌入不匹配**：ChromaDB 使用内置的 `all-MiniLM-L6-v2`（英文模型），未启用配置的 `text-embedding-v2`（中文模型）。这意味着中文文档的检索质量可能不是最优
- **简单分块**：字符级滑动窗口，不按句子/段落边界分割
- **无重排序**：检索结果直接按 ChromaDB 默认相关度排序，无 reranker

## 9. LangGraph Agent

### 9.1 ChatAgent（对话 Agent）

**图结构：**

```
START
  │
  ▼
┌──────────┐
│ retrieve │  ← 从 ChromaDB 检索相关知识
└────┬─────┘
     ▼
┌──────────────┐
│ match_presets│  ← 预设匹配（当前为空操作）
└────┬─────────┘
     ▼
┌──────────┐
│ generate │  ← 组装上下文 + 调用 LLM 生成回复
└────┬─────┘
     ▼
    END
```

**State 定义**（`ChatAgentState`）：

| 字段 | 类型 | 说明 |
|------|------|------|
| `messages` | `list[BaseMessage]` (with add_messages reducer) | 对话消息列表 |
| `knowledge_base_id` | `str` | 当前知识库 ID |
| `template_content` | `str` | 客服模版内容 |
| `conversation_id` | `str` | 会话 ID |
| `retrieved_docs` | `list[str]` | 检索到的文档块 |
| `preset_matches` | `list[dict]` | 预设对话匹配结果 |
| `final_response` | `str` | Agent 最终回复 |

**关键流程：**

1. `retrieve` 节点：调用 ChromaDB，获取 5 个最相关文档块
2. `match_presets` 节点：（空操作）实际匹配逻辑在路由层通过 SQL ILIKE 提前完成
3. `generate` 节点：组装 `SystemMessage(角色+模版+资料) + 历史消息 + 当前消息`，调用 LLM

**注意**：预设对话匹配在路由器中执行，**短路绕过整个 Agent**。如果关键词匹配成功，直接返回预设答案，不调用 LLM。

### 9.2 CopywritingAgent（文案生成 Agent）

**图结构：**

```
START
  │
  ▼
┌──────────┐
│ generate │  ← 填充产品信息 → LLM 生成文案
└────┬─────┘
     │
     ▼ _should_revise() → 始终返回 "done"
    END
```

**State 定义**（`CopywritingState`）：

| 字段 | 说明 |
|------|------|
| `product_name/type/features/price/promotion/audience/stock` | 产品信息输入 |
| `generated_copy` | LLM 生成的营销文案 |
| `revision_history` | 修订历史（预留，未使用） |

**提示模板**：结构化的中文电商文案 Prompt，要求包含产品标题、卖点提炼、详细描述、促销引导四部分，300-500 字，Markdown 格式。

### 9.3 Agent 调用方式

两个 Agent 都使用**单例模式**：

```python
_chat_agent = None

def get_chat_agent() -> ChatAgent:
    global _chat_agent
    if _chat_agent is None:
        _chat_agent = ChatAgent()
    return _chat_agent
```

在 FastAPI 路由中通过 `get_chat_agent()` 获取单例，然后调用 `agent.chat(...)`。

## 11. 前后端对接

### 11.1 开发环境

- 前端开发服务器：`http://localhost:3000`（Vite）
- 后端 API 服务器：`http://localhost:8000`（Uvicorn）
- Vite 配置了 **proxy**：所有 `/api/*` 请求代理到 `http://localhost:8000`

### 11.2 数据流示例

以"发送消息"为例：

```
1. 用户在 ChatPage 输入框输入 "这个衣服掉色怎么办"，点击发送
2. ChatPage.tsx handleSend():
   - setMessages(prev => [...prev, {role:'user', content:'...'}])   // 乐观更新
   - chatApi.send({ knowledge_base_id, template_id, message, conversation_id })
   - fetch('/api/chat', { method: 'POST', body: JSON.stringify(...) })
3. Vite Dev Server → proxy → http://localhost:8000/api/chat
4. FastAPI Router → chat.py → chat()
   - MemoryService.create_conversation() / get_conversation()
   - DialogService.search() → ILIKE 匹配预设 → 匹配则直接返回
   - MemoryService.load_history() → 加载历史消息 + 摘要
   - ChatAgent.chat() → retrieve → generate → LLM 回复
   - MemoryService.add_message() ×2 (user + assistant)
   - MemoryService.maybe_compress()
   - 返回 ChatResponse { reply, conversation_id }
5. Frontend ChatPage: setMessages(prev => [...prev, {role:'assistant', content: reply}])
6. React 重新渲染，显示新消息气泡
```

### 11.3 前端会话持久化

- `conversation_id` 存在 `localStorage` key `current_conversation_id`
- 页面刷新后，ChatPage 从 localStorage 读取 ID 并调用 `conversationApi.get(id)` 恢复消息
- "新建会话"按钮清除 localStorage 并重置消息列表

## 13. 已知局限与改进方向

### 13.1 已知局限

| 问题 | 影响 | 建议 |
|------|------|------|
| ChromaDB 使用默认英文嵌入模型 | 中文检索质量可能不佳 | 显式传入 `text-embedding-v2` 的嵌入函数 |
| 分块为简单字符滑动窗口 | 可能在句子中间截断 | 使用 LangChain 的 `RecursiveCharacterTextSplitter` |
| 无 reranker | 检索精度受限 | 引入 Cross-Encoder 重排序 |
| Redis 已配置但未使用 | 无缓存层，重复请求重复计算 | 添加对话缓存或限流 |
| 预设匹配使用 ILIKE 而非语义搜索 | 无法匹配同义表达 | 使用嵌入向量做语义匹配 |
| 无 token 预算管理 | 长对话可能超出 LLM 上下文窗口 | 动态追踪 token 数，超出时强制压缩 |
| 文案 Agent 的 Revise 节点为空 | 无法根据反馈修改文案 | 实现用户反馈环路 |
| 无用户认证/授权 | 所有 API 对外开放 | 添加 JWT 认证中间件 |
| 前端无路由 | URL 不反映当前页面 | 引入 React Router |

### 13.2 改进方向

1. **嵌入模型统一**：显式配置 ChromaDB 使用 DashScope 的 `text-embedding-v2`
2. **分块优化**：引入语义分块，按段落/句子边界分割
3. **Reranker**：使用 Cross-Encoder 对检索结果重排序
4. **缓存层**：利用 Redis 缓存常见问题的回复
5. **语义预设匹配**：将预设对话也向量化，用语义搜索替代 ILIKE
6. **动态 Token 管理**：监控上下文 token 数，超出时自动触发压缩
7. **用户系统**：添加多租户/多用户支持
8. **前端路由**：引入 React Router，支持 URL 导航和浏览器前进/后退
